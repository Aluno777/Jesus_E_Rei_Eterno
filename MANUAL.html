Docker no Windows

O RedShield roda completamente em Docker — sem Python, Django ou dependências na máquina host.

ℹ️
Requisitos: Windows 10 v2004 (build 19041+) ou Windows 11. CPU com VT-x / AMD-V. 4 GB RAM para o Docker.
WSL 2 + Docker Desktop
PowerShell — Administrador
# 1. Instalar WSL 2
wsl --install
wsl --set-default-version 2

# 2. Reiniciar o sistema

# 3. Baixar Docker Desktop em docker.com/products/docker-desktop
#    Marcar "Use WSL 2 instead of Hyper-V" na instalação

# 4. Verificar
docker --version
Docker version 27.x.x
docker compose version
Docker Compose version v2.x.x

Instalação e Primeiro Start

3 comandos para subir a aplicação completa.

CMD / PowerShell — pasta do projeto
# Entrar na pasta
cd redshield_django

# Build da imagem Docker (primeira vez ~2-5 min)
docker compose build

# Subir em background
docker compose up -d

# Ver logs ao vivo
docker compose logs -f
🚀
Aplicação rodando! Abra http://localhost:8000 no navegador.

30 Técnicas de Bypass de WAF

10 novas técnicas adicionadas na v3.0. Todas seguem a política anti-falso-positivo — ver seção 07.

01 — Null Byte SQL Injection
SQLi
Payload: 1%00' OR '1'='1 — Byte nulo quebra o token antes do payload para parsers C/C++.
02 — Double URL Encoding
SQLi/XSS
WAF decodifica uma vez; aplicação decodifica duas. Bypassa WAFs cloud sem double-decode.
03 — SQL Keyword Case Variation
SQLi
SeLeCt+UsEr()+FrOm+dUaL — Bypassa regras case-sensitive.
04 — SQL Inline Comment Fragmentation
SQLi
SE/**/LE/**/CT — Comentários fragmentam tokens de palavras-chave.
05 — MySQL Conditional Comment Evasion ✨ NEW
SQLi
/*!50000SELECT*/ — WAF vê comentário; MySQL executa como SQL válido.
06 — Unicode UTF-8 IIS Notation
SQLi/XSS
%u0053%u0045... — Notation %uXXXX do IIS. Alta efetividade em ASP.NET legado.
07 — XSS HTML Entity Decimal Encoding
XSS
Tags como entidades decimais. Browser renderiza e executa; WAF não reconhece.
08 — XSS SVG onload Backtick Bypass
XSS
<svg/onload=alert`1`> — Sem espaço, backticks no lugar de parênteses.
09 — XSS JavaScript Protocol href ✨ NEW
XSS
<a href=javascript:alert(1)> — Sem script tag; bypassa filtros focados em <script>.
10 — XSS IMG onerror Polyglot ✨ NEW
XSS
<img src=x onerror=alert(1)//> — IMG com src inválido sem script tag.
11 — Path Traversal Double URL Encoding
LFI
..%252F..%252Fetc%252Fpasswd — App decodifica duas vezes.
12 — Path Traversal Unicode Overlong Slash
LFI
..%c0%af..%c0%afetc — Barra como UTF-8 inválido. CVE-2001-0131.
13 — Path Traversal Encoded Dot-Dot ✨ NEW
LFI
%2e%2e%2f%2e%2e%2fetc%2fpasswd — Dot-dot-slash completamente codificado.
14 — HTTP Header IP Spoof via X-Forwarded-For
IP Bypass
X-Forwarded-For: 127.0.0.1 — Apps sem validação de proxies.
15 — HTTP Header IP Spoof via X-Real-IP ✨ NEW
IP Bypass
X-Real-IP: 127.0.0.1 — Alternativa XFF para ambientes Nginx.
16 — Chunked Transfer Encoding Fragmentation
WAF Evasion
Fragmenta payload em chunks. WAFs sem reassembly perdem o contexto.
17 — HTTP Request Smuggling TE-CL
Request Smuggling
Conflito Transfer-Encoding / Content-Length. Engana proxy sobre limites.
18 — HTTP Request Smuggling CL-TE ✨ NEW
Request Smuggling
Variante CL-TE. Frontend usa CL; backend usa TE — janela de injeção.
19 — JSON Content-Type SQLi Bypass
SQLi/NoSQLi
Payload em JSON. WAFs para form-urlencoded ignoram o body completamente.
20 — Multipart Boundary Injection ✨ NEW
WAF Evasion
Payload no nome do campo multipart. Parsers WAF raramente inspecionam esta área.
21 — SSRF via DNS Rebinding
SSRF
DNS controlado rebinda para IP interno após check do WAF.
22 — SSRF via IPv6 Loopback ✨ NEW
SSRF
http://[::1]/admin — Bypass de filtros baseados apenas em 127.0.0.1.
23 — SSRF via Decimal IP Notation ✨ NEW
SSRF
http://2130706433/ — 127.0.0.1 como inteiro decimal.

GraphQL

Política Anti-Falso-Positivo

Uma das principais melhorias da v3.0. Resultados falso-positivos comprometem a credibilidade do relatório e o tornam inutilizável profissionalmente.

// REGRAS DE CLASSIFICAÇÃO — engine.py → test_waf_bypasses()

BYPASS → HTTP 200..299 E body NÃO contém padrões de bloqueio WAF
BLOQUEADO → HTTP 403, 406, 429, 503, 418
BLOQUEADO → HTTP 200..299 com body contendo: "access denied", "request blocked",
"cloudflare", "incapsula incident", "waf blocked", etc.
NOT_FOUND → HTTP 404 — endpoint /search inexistente no alvo — NÃO É BYPASS
REDIRECT → HTTP 301, 302, 307, 308 — redirecionamento — NÃO É BYPASS
PARCIAL → HTTP 400, 405, 500, 502 — resposta anômala, requer análise manual
ERROR → Timeout ou falha de conexão
⚠️
Por que 404 não é bypass?
Um 404 significa que o endpoint /search?q=payload simplesmente não existe naquele servidor. O WAF nem chegou a ser exercitado. Reportar isso como "bypass" seria enganar o cliente — o relatório perderia toda a credibilidade profissional.
⚠️
Por que 301/302 não é bypass?
Um redirect indica que o servidor está redirecionando a requisição — frequentemente para uma página de bloqueio do próprio WAF, ou para HTTPS. Não há evidência de que o payload chegou a ser executado ou avaliado pela aplicação.
